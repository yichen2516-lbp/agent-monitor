# Agent Monitor Phase 5 规划

## 定位

Phase 5 不再是“结构性重构”，而是 **产品体验与可观察性增强阶段**。

前面 Phase 1~4 已完成的重点：
- 拆薄入口与路由层
- 拆分前端 monitor 脚本
- 拆分后端 monitor-store 内核
- 补齐最小自动化回归测试与文档

因此，Phase 5 的目标不是继续大拆，而是：

> **在现有结构已经清晰的前提下，把 Agent Monitor 做得更顺手、更稳、更像一个成品。**

---

## 目标

### 总目标
围绕三个方向推进：
1. **更好看懂**：更容易理解当前发生了什么
2. **更好定位**：更快发现错误、慢请求、异常模式
3. **更好操作**：筛选、查看、跳转、排查路径更顺手

### 约束
- 不破坏当前已稳定的页面与 API 主行为
- 不引入重型前端框架
- 尽量做增量优化，避免再次进入大规模结构改造
- 每项优化都应可单独交付

---

## 建议拆成 5 个子方向

## P5-A：监控视图增强

### 目标
让主监控页更适合“盯盘”和“排错”，不只是活动列表滚动。

### 候选项

#### 1. Agent 维度视图
当前：所有活动混在一个 feed 中
目标：支持按 agent 快速聚焦

可选实现：
- agent 快速 tab / segment
- agent 独立摘要卡片
- 每个 agent 最近活动时间显示

#### 2. Session 维度视图
当前：session 只作为活动卡片里的一个字段
目标：支持按 session 追踪完整上下文

可选实现：
- session 筛选器
- session drill-down 面板
- session 最近 N 条活动聚合展示

#### 3. 类型视图增强
当前：tool / reply / thinking / cron 仅靠筛选器区分
目标：支持更清楚的信息层级

可选实现：
- 类型分组显示
- thinking 默认折叠
- tool call / tool result 成对视觉展示

### 预期收益
- 更快锁定“哪个 agent / 哪个 session / 哪类事件”有问题
- 降低活动流刷屏时的信息噪音

---

## P5-B：错误与异常定位增强

### 目标
让 Agent Monitor 从“看日志”升级到“快速识别异常模式”。

### 候选项

#### 1. Error 聚合增强
当前：已有基础 error aggregate
可继续增强：
- 按 tool + agent + error signature 聚合
- 展示最近发生时间 / 次数 / 首次时间
- 支持一键展开组内原始条目

#### 2. 慢调用视图
当前：只有 summary card 统计 Slow Calls
可继续增强：
- slow calls 列表
- duration 排序
- duration 阈值可配置

#### 3. 失败请求专用过滤器
增加：
- 仅显示 `exitCode > 0`
- 仅显示 cron error
- 仅显示带 `toolError` 的事件

#### 4. 错误详情增强
在 detail drawer 中补更多上下文：
- toolStatus
- stopReason
- source file
- usage/token 结构化展示
- 原始 event 片段（如果可用）

### 预期收益
- 更适合用来做第一现场排查
- 减少手动翻日志文件的频率

---

## P5-C：详情交互与可读性优化

### 目标
让单条活动的阅读体验更好，降低信息挤压感。

### 候选项

#### 1. 卡片信息层级优化
- 时间 / agent / session 视觉层级更清楚
- model / token / duration / exitCode tag 更统一
- description 过长时支持折叠/展开

#### 2. Drawer 升级
当前 drawer 偏原始 JSON 展示
建议升级为：
- 概览区（关键字段）
- 原始 JSON 区（可折叠）
- source path 区
- usage 区

#### 3. 快捷复制能力
增加按钮：
- 复制 event JSON
- 复制 source path
- 复制 session id / agent 名称

#### 4. 文本显示优化
- 长文本自动换行策略更优
- markdown/代码片段显示更清楚
- 大段 thinking/reply 不要把列表撑爆

### 预期收益
- 单条事件更易读
- 调试时复制信息更方便

---

## P5-D：Workspace Browser 产品化增强

### 目标
把 workspace 页面从“能看”升级到“好用”。

### 候选项

#### 1. 文件搜索
- 按文件名搜索
- 支持当前 agent workspace 内快速过滤

#### 2. 目录状态记忆
- 记住展开/折叠状态
- 记住当前 agent tab

#### 3. 文件查看增强
- 更好的 markdown 显示
- 代码文件简单高亮（后期可选）
- 大文件提示与截断说明

#### 4. 导航增强
- 上一个 / 下一个文件
- 面包屑更易点
- 更好的移动端返回体验

### 预期收益
- workspace 页更像一个轻量浏览器，而不是只读 demo

---

## P5-E：性能与可观察性增强

### 目标
在功能稳定基础上，把“更稳、更省、更可追踪”补上。

### 候选项

#### 1. API 指标增强
- `/api` 请求计数更结构化
- 增加 error count / avg latency / p95（后续可选）
- 服务启动后累计指标面板

#### 2. 前端渲染优化
- 降低无意义重绘
- 大量活动下的渲染性能优化
- 过滤器变更时减少 DOM 抖动

#### 3. Polling 策略优化
- 根据页面活跃状态降频
- 页面失焦时降低刷新率
- 可选的“暂停刷新”按钮

#### 4. 向 WebSocket 演进的预研
不是现在必须做，但可以记录方向：
- 先保留 polling
- 后续评估 websocket 推送是否值得

### 预期收益
- 更适合长期挂着监控
- 页面更稳，资源消耗更低

---

## 建议优先级

### P5 优先级排序（建议）

#### 第一优先级（最值得做）
1. **P5-B 错误定位增强**
2. **P5-C 详情与可读性优化**

原因：
- 这两项直接提升排错效率
- 对 Agent Monitor 的实际价值最大
- 不需要大改架构

#### 第二优先级
3. **P5-A 监控视图增强**
4. **P5-D Workspace Browser 增强**

原因：
- 提升整体产品完成度
- 但不是最急迫的“排错刚需”

#### 第三优先级
5. **P5-E 性能与可观察性增强**

原因：
- 重要，但适合在前面几个方向稳定后再做

---

## 建议的执行方式

### 不建议
- 一口气做完整个 Phase 5
- 把多个方向混成一个大 PR
- 再次大规模改结构

### 建议
把 Phase 5 当作一个 backlog，按小专题推进：

#### 方案 A：按价值推进
1. 先做错误定位增强
2. 再做详情 drawer 升级
3. 再做 workspace 增强

#### 方案 B：按页面推进
1. 先把 monitor 主页做细
2. 再做 workspace 页
3. 最后再做性能/指标优化

我更建议 **方案 A**，因为更贴近实际使用价值。

---

## 可直接开做的第一批任务（建议）

### Task 1：增强错误聚合卡片
- error group 展示次数 / 最近一次时间
- 支持展开组内项
- 支持按 tool + agent + exitCode 聚合

### Task 2：升级 detail drawer
- 分成 summary + raw JSON + source path 三段
- 增加 copy 按钮
- usage 和 duration 单独展示

### Task 3：加入 slow calls 视图
- 点击 Slow Calls summary card 可过滤慢调用
- 或增加 quick button: `Slow Only`

### Task 4：Workspace 增加文件搜索
- 只做文件名搜索，先别做全文搜索

---

## 验收标准

Phase 5 不适合用“文件有没有拆完”来验收，而应该用用户价值来验收：

### 好的验收方式
- 出错时能否更快定位问题
- 活动流是否更容易读
- 详情信息是否更完整
- workspace 是否更顺手
- 长期开着是否更稳

### 不好的验收方式
- 只是代码更多了
- UI 更花了
- 做了很多但没有提升排错效率

---

## 一句话结论

**Phase 5 = 产品增强 backlog，不是必须连续做完的重构阶段。**

最值得优先做的是：
1. 错误定位增强
2. 详情 drawer 升级
3. 再考虑 workspace 和性能优化
